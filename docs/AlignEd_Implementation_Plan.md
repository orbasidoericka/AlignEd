## 1. Executive Overview & Architecture

### Technical approach in one paragraph

AlignEd ships as a **mostly-static, anonymous-first** Next.js 16 application. Reference content ‚ÄĒ degree programs and careers ‚ÄĒ is public-read data rendered on the server and edge-cached, so the pages Filipino students hit on low-end phones over variable connections are cached HTML. The profile capture and RIASEC assessment run **entirely client-side** (the existing Zustand + localStorage store), which means the platform's only write paths are (a) one anonymous `assessments` insert per completed assessment and (b) the `send-results` email Edge Function. That architecture makes the PRD's 5,000-concurrent-sessions requirement land on the edge cache rather than on Postgres.

### System layers (Input‚ÄďProcess‚ÄďOutput, mapped to real components)

- **Input:** static TypeScript question bank ‚Üí client-side profile + answers in `useAssessmentStore` (already implemented with localStorage persistence at `apps/web/src/store/useAssessmentStore.ts`); programs/careers seeded into Supabase Postgres from `data/courses.csv` and `data/careers.csv`.
- **Process:** pure-TS scoring library (`src/lib/riasec/scoring.ts`) derives the 3-letter Holland code with deterministic R‚ÜíI‚ÜíA‚ÜíS‚ÜíE‚ÜíC tie-breaking (PRD FR-4); pure-TS strand-validation module (`src/lib/riasec/strand-validation.ts`) applies the canonical strand‚Üíletters matrix and overlap rule (PRD FR-5); the existing `match_courses(p_code)` SQL RPC (migration `0001_init.sql`) ranks programs by shared Holland letters (exact = 3 shared, partial = 2).
- **Output:** client-computed results dashboard (code, radar, verdict, careers, programs); client-side print-to-PDF export; emailed results via Resend.

### Tech stack

| Layer | Choice | Notes |
|---|---|---|
| Frontend | **Next.js 16.2.5** App Router, React 19, TypeScript | Breaking changes vs Next 15 ‚ÄĒ read `docs/Next16_Delta_Notes.md` and the bundled docs at `node_modules/next/dist/docs/` before framework work. |
| Styling | **Tailwind v4** (`@tailwindcss/postcss`) + `tw-animate-css` | Design tokens live in `@theme` inside `globals.css`; vendored MagicUI/Aceternity keyframes ported there once. No `tailwind.config.ts`. |
| Components | **shadcn/ui (Base UI) + MagicUI + Aceternity + Efferd**, all vendored via the `shadcn` CLI registry system into `src/components/{ui,magic,aceternity,blocks}` | Copy-paste ownership: every animated component can be performance-tuned, reduced-motion-gated, and themed locally. |
| Motion | framer-motion 12 (`motion/react`) behind a single `LazyMotion` provider | One orchestration layer; tree-shaken features; global `useReducedMotion` gate. |
| Backend | **Supabase** ‚ÄĒ Postgres, RLS, Edge Functions; **Auth product deliberately unused** (PRD ¬ß3 Non-Goals) | RLS: public-read on reference tables, insert-only on `assessments`, zero read-back. |
| Email | **Resend** called from the `send-results` Supabase Edge Function | DKIM/SPF for deliverability; raw email addresses never persisted (PRD FR-9). |
| Infra | Vercel (web) + Supabase Cloud (data); GitHub Actions CI; Sentry + PostHog + Vercel Analytics | Staging and production are separate Vercel + Supabase environments. |

### Database plan

Migration `0001_init.sql` already provides `careers`, `courses`, `course_careers`, anonymous `assessments`, indexes on `riasec_tag`, RLS policies, and the `match_courses` RPC.

**Changes required to match the PRD:**

1. **New migration `0002_assessment_profile.sql`:** add `grade_level` (check-constrained to `11`/`12`), `strand` (check-constrained to the 7 canonical values), and `school` (nullable text, length ‚Č§ 120) to `assessments`. Insert-only RLS unchanged.
2. **Descope universities:** migration `0002_universities.sql` and the `data/universities.csv` / `data/university_courses.csv` inputs are out of scope per the PRD (no directory, no university mapping). Cleanup task 1.2 removes them so the schema and seed pipeline contain nothing the product does not use.
3. `supabase/seed.sql` is generated by the schema-validated CSV pipeline (`scripts/build-seed.mjs`) from `data/courses.csv`, `data/careers.csv`, `data/course_careers.csv` only.

## 2. UI/UX Component Strategy

Design language and page-level wireframes live in `docs/AlignEd_Design_Vision.md`. Division of labor across the four libraries: **Efferd** supplies page-level structure (header, footer, hero, FAQ); **Aceternity** supplies visual identity moments (hero background, bento, spotlight cards); **MagicUI** supplies micro-interactions and gamified feedback (confetti, sparkles, progress ring); **shadcn/ui** supplies every foundational control ‚ÄĒ it is the accessibility floor.

| Feature (PRD ref) | Component mapping |
|---|---|
| **Landing page** (FR-1) | **Efferd**: Hero block + Header/Nav + Footer + FAQ skeleton. **Aceternity**: *Hero Highlight* behind the headline; *Bento Grid* for the four journey steps; *Infinite Moving Cards* testimonials. **MagicUI**: *Animated Shiny Text* eyebrow; *Shimmer Button* on the journey-aware CTA; *Number Ticker* stats. **shadcn**: `Sheet` mobile nav, `Button`. |
| **Profile Setup** (FR-2) | **shadcn**: `RadioGroup` (grade level ‚ÄĒ two large cards), `Select` or segmented `ToggleGroup` (strand ‚ÄĒ seven options with short descriptions), `Input` with counter (school, optional), `Button` (disabled until valid), `Tooltip` explaining why strand is asked. Runs inside the same Focus Mode shell as the quiz. |
| **RIASEC assessment** (FR-3) | **shadcn**: `Progress`, `Card`, `RadioGroup` (Likert as five tappable cards), `Button`, `AlertDialog` on exit attempts. **MagicUI**: *Animated Circular Progress Bar* step counter; *Blur Fade* question entrance; *Confetti* on completion. **Framer Motion**: `AnimatePresence` slide transitions. **State**: existing `useAssessmentStore` (kept ‚ÄĒ it already handles re-answer score correction and localStorage resume), extended with profile fields. |
| **Results ‚ÄĒ code reveal** (FR-4) | **MagicUI**: *Sparkles Text* Holland-code reveal; *Animated Gradient Text* trait name. **shadcn**: `Chart` (Recharts `RadarChart`) six-trait radar wrapped in Aceternity *Card Spotlight*. |
| **Results ‚ÄĒ Strand Validation** (FR-5) | **shadcn**: verdict `Badge` (Aligned / Partially Aligned / Misaligned / Aligned‚ÄĒFlexible) + `Card` with shared-letter evidence chips and insight copy; **MagicUI**: *Border Beam* frames the verdict card (it is the product's headline feature). |
| **Results ‚ÄĒ careers & programs** (FR-6, FR-7) | **Aceternity**: *3D Card Effect* for the #1 program match; *Card Hover Effect* grid for the rest and for careers. **shadcn**: `Badge` match strength ("Exact match" / "Strong match") derived from the RPC's `match_score`. No university links anywhere. |
| **PDF export** (FR-8) | Print-optimized results view (dedicated print stylesheet; interactive chrome hidden via `print:` variants) triggered through `window.print()`. A4 single column, grayscale-legible. |
| **Email-my-results** (FR-9) | **shadcn**: `Dialog` + `Form` (zod validation) + `Input` + consent `Checkbox` (Data Privacy Act copy) + `Sonner` toasts. **MagicUI**: *Pulsating Button* send CTA. Honeypot field included. |

**State management strategy:** Server Components own all reference data ‚ÄĒ no client-side global state mirrors server data. Zustand is scoped to the profile + assessment session store. There is no other client state.

**Animation orchestration:** one `LazyMotion` provider at the root; every vendored animated component consumes a shared `useReducedMotion` gate so `prefers-reduced-motion` collapses nonessential motion globally. Scroll-triggered animations fire on viewport entry, never on mount.

## 4. QA & Testing Protocol

**Unit (Vitest ‚ÄĒ already configured).** Highest-value targets: the scoring library (property-based tests ‚ÄĒ every possible answer set yields a valid 3-letter code; ties resolve by R‚ÜíI‚ÜíA‚ÜíS‚ÜíE‚ÜíC; re-answering never corrupts totals ‚ÄĒ the store's subtraction logic in `setAnswer` gets explicit coverage), the strand-validation module (full 7-strand √ó verdict matrix, GAS special case), and profile validation. Coverage target ~90% on `src/lib/riasec/`.

**Integration.** React Testing Library with a mocked Supabase client for component behavior; plus **RPC contract tests** that run `supabase start` in CI against real migrations + seed and assert `match_courses` ranking semantics (exact-before-partial, alphabetical tiebreak) ‚ÄĒ so the SQL and the TypeScript expectations can never silently drift.

**E2E (Playwright ‚ÄĒ already installed).** Four journeys: (1) land ‚Üí profile ‚Üí complete full assessment ‚Üí results (code, verdict, careers, programs all present) ‚Üí email dialog with Resend mocked; (2) mid-quiz refresh ‚Üí resume; (3) guard rails ‚ÄĒ `/results` without completion redirects, `/assessment` without profile redirects, invalid email blocked inline, consent unchecked blocks send; (4) print view ‚ÄĒ the PDF route renders all four result sections with chrome hidden.

**Visual regression.** Playwright `toHaveScreenshot` per key page (landing, profile, quiz, results, print view) in **light and dark themes**, captured under `reducedMotion: 'reduce'` emulation so the global motion gate freezes every animation deterministically. Snapshots generated and compared on Linux CI only. A separate "animation smoke" spec runs with motion enabled and asserts every animated component mounts without console errors.

**Performance budgeting.** Lighthouse CI asserts ‚Č• 90 on every PR (mobile emulation, throttled CPU). Route-level JS budgets from `next build` output: landing ‚Č§ ~180 KB gzipped. LCP < 2.5 s on throttled mid-tier mobile. Canvas-based background effects measured under CPU throttling ‚ÄĒ if any blows the budget, it is swapped for its CSS-gradient variant. Images through `next/image`; fonts subset and self-hosted.

**UAT.** Scripted sessions with Grade 11 and Grade 12 students across strands, validating the "Empathetic Guidance" thesis and specifically the emotional reception of *Misaligned* verdicts, paired with PostHog funnels (per-question drop-off, results ‚Üí export conversion) as the quantitative companion.

## 5. Risk Mitigation & Edge Cases

| Risk | Impact | Mitigation |
|---|---|---|
| **Next 16 breaking changes** vs Next 15 assumptions in registry docs and habit | Subtle build/runtime failures | `docs/Next16_Delta_Notes.md` maintained; versions pinned; CI build gate on every PR. |
| **Registry components assume Tailwind v3 + Radix** while the repo is Tailwind v4 + Base UI | Vendored components break or double-bundle | Vendor the full set in Phase 1 (task 1.4) to surface incompatibilities immediately; keep all fixes local to `components/{magic,aceternity}`. |
| **Animation weight degrades low-end mobile** (the primary audience) | Lighthouse < 90; students bounce | Global reduced-motion gate, `LazyMotion`, viewport-triggered animation only, CSS fallbacks for canvas effects, throttled-CPU perf tests in CI. |
| **Misaligned verdicts land badly** with students or educators | Trust damage, negative word of mouth | Insight copy reviewed in UAT (task 5.5); matrix versioned in one module with documented rationale; verdicts framed as guidance, never judgment. |
| **Email abuse / Data Privacy Act exposure** (psychometric results + email = sensitive personal information) | Legal and trust damage | No accounts; raw email never stored (hash only, for rate limiting); explicit consent checkbox with DPA copy; public privacy notice; honeypot + per-IP rate limits; DKIM/SPF via Resend. |
| **Static program/career data goes stale** | Loss of user trust | Data edits via versioned CSV ‚Üí seed pipeline shipped as reviewed PRs. |
| **Seed-data research slips** | Phase 3 blocked | Content track starts day one as a parallel workstream with a schema-validated CSV pipeline; Phase 3 can proceed on partial data. |
| **5,000 concurrent assessment sessions** | Degraded response times | Assessment is fully client-side; reference pages are edge-cached; the only write is an anonymous insert. k6 sanity test in Phase 5. |
